# v0.12.0 — Match Engine 2.0 multiárea

## Objetivo

A v0.12.0 deve substituir a ideia de match baseado principalmente em cobertura de palavras por uma engine multiárea, explicável, ponderada e baseada em evidências.

A IA deve ajudar a classificar e explicar. O código deve calcular.

## Problema

Um match simples por keyword funciona para triagem, mas falha quando:

- a vaga é de área não técnica;
- o requisito é registro profissional;
- a vaga usa sinônimos;
- a competência é transferível;
- o currículo tem evidência em projeto ou portfólio;
- a senioridade está implícita;
- a vaga mistura obrigatório e desejável;
- o requisito é equipamento, norma ou ambiente.

## Princípio

```text
Match Score não é porcentagem de palavras. É compatibilidade entre requisitos, evidências, contexto e preferências.
```

## Inputs

A engine deve receber:

- `ResumeExtractionOutput`;
- `JobExtractionOutput`;
- preferências do usuário;
- evidências da memória;
- evidências de GitHub/portfólio;
- histórico do tracker;
- resultado ATS;
- sinais de risco;
- domínio profissional.

## Outputs

A engine deve produzir:

- Match Score;
- ATS Score;
- Opportunity Fit Score;
- Risk Score;
- Evidence Strength Score;
- Confidence Score;
- recomendação;
- explicação;
- gaps;
- ações recomendadas;
- campos que exigem revisão.

## Fórmula base sugerida

```text
required_requirements: 25%
domain_specific_competencies: 20%
education_and_credentials: 15%
experience_evidence: 15%
work_model_location: 10%
tools_equipment_systems: 5%
soft_skills: 5%
risk_adjustment: 5%
```

A fórmula deve ser configurável por domínio.

## Exemplo de ajuste por área

### Enfermagem

- credencial profissional: peso alto;
- setor de atuação: peso alto;
- disponibilidade de escala: peso médio;
- soft skills: peso médio;
- GitHub: peso nulo.

### Arquitetura

- portfólio visual: peso alto;
- ferramentas: peso alto;
- projeto executivo: peso alto;
- registro profissional: conforme vaga;
- GitHub: peso baixo ou nulo.

### Software

- stack obrigatória: peso alto;
- arquitetura/projetos: peso alto;
- testes e GitHub: peso médio/alto;
- formação: depende da vaga;
- portfólio técnico: peso alto.

### Pedagogia

- etapa de ensino: peso alto;
- BNCC: peso médio/alto;
- inclusão: conforme vaga;
- experiência em sala: peso alto;
- portfólio acadêmico: peso médio.

## Status de match por requisito

Cada requisito deve virar um item:

```json
{
  "requirement": "AutoCAD",
  "importance": "required",
  "status": "matched",
  "evidence": ["Currículo: habilidades técnicas"],
  "score_contribution": 8.5,
  "confidence": 0.88
}
```

Status possíveis:

- `matched`;
- `partial`;
- `missing`;
- `transferable`;
- `unclear`;
- `not_applicable`.

## Knockout gaps

Alguns gaps devem limitar o score.

Exemplos:

- registro profissional obrigatório ausente;
- diploma obrigatório ausente;
- localidade incompatível quando presencial obrigatório;
- senioridade muito acima;
- idioma obrigatório sem evidência;
- requisito legal obrigatório ausente.

Regras de trava:

```text
Se houver knockout gap real, Match Score máximo sugerido: 45.
Se houver dois ou mais knockout gaps, Match Score máximo sugerido: 30.
Se o gap puder ser apenas falta de informação, pedir revisão antes de travar.
```

## Competências transferíveis

Competências transferíveis devem somar, mas não substituir requisitos críticos.

Exemplo:

```text
Experiência com atendimento em enfermagem pode ajudar em healthtech,
mas não comprova experiência em produto, dados ou desenvolvimento.
```

## Explicação obrigatória

Todo score deve responder:

- o que combinou;
- o que faltou;
- o que é parcial;
- o que é transferível;
- o que é gap crítico;
- o que pode ser melhorado no currículo;
- o que não deve ser inventado;
- qual a confiança da análise.

## Sugestões seguras

A engine deve classificar ações:

- `highlight_existing_evidence`;
- `rewrite_for_clarity`;
- `add_if_true`;
- `learn_or_build_gap`;
- `avoid_or_apply_with_caution`.

## Testes

Criar testes para:

- match forte de TI;
- match fraco por senioridade;
- enfermagem com COREN ausente;
- pedagogia com BNCC presente;
- engenharia civil com AutoCAD e obra;
- arquitetura com portfólio ausente;
- cybersecurity com labs como evidência parcial;
- competência transferível;
- requisito obrigatório vs desejável;
- score travado por knockout gap;
- confidence baixo pedindo revisão.

## Critério de pronto

A v0.12.0 estará pronta quando:

- o score não depender apenas de keywords;
- requisitos forem classificados;
- domínios alterarem pesos;
- gaps críticos travarem score quando necessário;
- competências transferíveis aparecerem;
- evidências forem citadas;
- a IA não definir score final sozinha;
- exemplos multiárea passarem nos testes;
- UI explicar o resultado de forma clara.
