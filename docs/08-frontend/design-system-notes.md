# Design system notes

Este documento não define um design fechado. Ele orienta um produto maduro, legível e
profissional.

## Direção visual

- SaaS-like, limpo e focado em tomada de decisão.
- Priorizar informação escaneável em vez de decoração.
- Usar hierarquia forte para score, confidence, ATS e próximos passos.
- Evitar telas que pareçam landing page quando a pessoa já está usando o app.
- Dar espaço para gráficos e tabelas no Kanban/Application Intelligence.

## Componentes esperados

- cards KPI;
- badges para status, fonte e confidence;
- tabs para Match, ATS, Tailor e Evidence;
- tabelas ranqueadas;
- gráficos de barras, funil, donut e linha temporal;
- kanban com cards compactos;
- editor de texto para currículo/vaga;
- callouts de segurança para itens sem evidência.

## Tom de interface

O produto deve ser direto, técnico e cuidadoso:

- "Evidência encontrada" em vez de "você domina";
- "Adicionar somente se for verdadeiro" em vez de "adicione ao currículo";
- "Gap crítico" quando registro, formação ou certificação não aparecem;
- "Baixa confiança" quando há pouca evidência.

## Acessibilidade

- contraste suficiente;
- foco visível;
- navegação por teclado;
- textos de erro claros;
- gráficos acompanhados de tabelas ou labels;
- não depender só de cor para status.
